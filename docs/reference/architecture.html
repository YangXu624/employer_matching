System architecture

Employer Match pipeline

This pipeline takes text job descriptions from employers, and automatically assesses how much employers value each of the six POG competencies, which is represented by scores that add up to 100. It then uses these scores to determine which students are most suitable for the employer.

Architecture overview

The pipeline is broken into four phases:

Phase 0 Setting up the rubrics database

Write a description for each POG competency at levels 1 through 5 (30 descriptions total). Embed these into vectors and store them.

Phase 1 Vector similarity

Take the employer's JD, break it into pieces, embed each piece, and measure how similar each piece is to each rubric level.

Phase 2 Scoring

For each POG, pick the level with the highest similarity. Combine level × similarity into an intermediate score, then normalise to 100.

Phase 3 Matching

Take each student's POG scores, weight them by the employer's scores, and rank students by suitability.

In short: we run the JD text against our own rubrics for each POG to see how much the employer values each skill.

Phase 0 Setting up the rubrics database

For each POG, we define what level 1 (least important) through level 5 (most important) looks like. With 6 POGs, that is 30 descriptions in total.

These 30 descriptions are then embedded using an embedding model into vectors (arrays of numbers that capture the meaning of the text), which are stored as a reference to compare against.

(In the actual implementation, these vectors are saved to a disk cache as JSON files under .cache/employer_match/, keyed by the model name and a hash of the descriptions. It is a cache, not a queryable database — no index or query language.)

Phase 1 Vector similarity

We take the employer's JD and break it into smaller pieces (sentences) actual: split on .?! boundaries. Each piece is embedded using the same embedding model from Phase 0, producing one vector per piece.

We then measure how similar each JD piece is to each of the 30 rubric level vectors using cosine similarity. The result is 30 similarity scores — 5 for each POG.

This gives us, for each of the 6 POGs, one peak similarity and a matched level.

Phase 2 Scoring

Within each POG, we pick the level with the highest similarity. This gives us a matched level (1–5) for each POG.

But level alone isn't enough. Imagine two POGs both matched at level 4, but one has a similarity of 20% while the other has 90%. The second is clearly more important to the employer. So we compute an intermediate score by multiplying the matched level by the similarity:

intermediate = level × similarity actual: level × max(similarity − calibration_baseline, 0)

The calibration baseline defaults to 0.0, so in practice it reduces to level × similarity.

Finally, we sum the intermediate scores from all 6 POGs and normalise them to add up to 100. These are the final weights — the employer's POG profile.

Phase 3 Matching

In this phase we compute a suitability score for each student using a weighted average. This involves two inputs:

  • Student POG scores — taken from the pipeline built by Rakshana (stored in data/candidates.csv). Each student has a score (0–100) in each of the 6 POGs.
  • Employer weights — the final POG scores from Phase 2, which serve as the weights in the weighted average.

The suitability score is calculated as:

suitability = ∑(student_POG_score × employer_weight) / total_weight

Students are then ranked by their suitability score, and the top few are presented to the employer.

Two options: To use an LLM or not?

An optional LLM check (powered by Gemini 2.5 Flash) runs alongside the vector pipeline. It re-scores the JD against the same rubric using natural language understanding, giving the user a comparison view. The results are shown side by side on the radar chart.

Pros

  • Can capture semantics and sentiment that pure vector matching misses — for example negation ("technical skill is not necessary") might be matched as a requirement by the embedding-based pipeline but correctly identified as an exception by the LLM

Cons

  • Cost. For a small number of JDs it is negligible — running a hundred might cost only ~$1. But it scales linearly.
  • Not deterministic. LLMs do not follow a fixed algorithm for generation, so repeated runs on the same JD can give slightly different results.

Alternative: local LLM (Ollama)

No API call costs, but requires running a local model, which can be heavy on system resources and slower. Not currently implemented.

Recommendation: use paid API calls

Stick with paid API calls (Gemini 2.5 Flash) for LLM scoring, just like how we generate the student POG passports. The cost is negligible at this scale — running a hundred JDs costs roughly $1. This keeps the implementation simple, the speed fast, and avoids the complexity of running and maintaining local models. The deterministic embedding pipeline remains the default; the LLM API is only called when the user explicitly clicks "LLM Check," so costs are entirely optional and usage-controlled.

Things to be done if proceeding LLM-free

Rubrics have to be updated

The quality of the vector similarity approach depends heavily on how well the rubric level descriptions discriminate between adjacent levels. The current rubric may need iteration to improve separation, especially for levels that currently produce close similarity scores.

What is our benchmark for accuracy?

We need a labelled dataset of JDs with human-assigned POG scores to measure against. Without this, we cannot quantify whether a change to the rubric or scoring algorithm is an improvement. One possible approach: compare against LLM output as a proxy baseline, treating the LLM as a "reasonable human" reference point, while acknowledging its own limitations. A better approach is to collect actual employer feedback on whether the suggested weights match their priorities.
