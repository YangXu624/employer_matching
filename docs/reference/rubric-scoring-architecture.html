System architecture · working draft

Automated rubric scoring

A zero-friction pipeline that extracts job requirements directly from existing JDs, matching them against Student Passports to find the highest-fit candidates.

The Problem

Employers are reluctant to manually input detailed competency requirements or fill out complex forms just to use a new hiring platform. This "onboarding friction" often prevents high-quality roles from being listed, as the manual effort required is perceived as too high for a specialized student pool.

The Solution: Automated Extraction

We eliminate manual entry by automatically extracting the importance of each competency directly from the employer's existing Job Description. By turning their plain-text JD into a structured "Employer Profile," we make SpeakHire incredibly easy to adopt. Employers simply "paste and go," with the option to fine-tune the AI's first guess if they choose.

To ensure this remains scalable and affordable, we implemented a cost-saving toggle for the LLM-check mode. It can be switched off to use lightweight embedding math for bulk processing, and only turned on when high-precision reasoning or negation-checking is required for a specific role.

The idea in one line

We turn a written rubric into a scoring engine for job descriptions, generating a weighted "Employer Profile" that we then intersect with "Student Passports" to find the highest-fit matches.

The architecture at a glance

How it works

01 Author the rubric once, up front

We write a rubric for each criterion — like the one in the brief, with fuller descriptions per box. Each of the five levels (5, 4, 3, 2, 1) gets a written description of what that level looks like. This is our grading standard, made explicit. The richer and more distinct each level's wording, the sharper the matching that follows.

02 Score against the rubric no AI · runs always

The input and each level description are turned into embeddings — vectors that capture meaning. We compute the cosine similarity between the input and all five levels; the level with the highest similarity wins, and its number becomes the score. This step is deterministic, cheap, instant at scale, and fully auditable — every score points back to the exact level it matched.

03 Optional LLM check a toggle

A switch, not a permanent dependency. Off, we ship the embedding scores as they are, at near-zero cost per item. On, the LLM reviews each score and corrects what plain matching can't — chiefly negation ("communication is nice but not required") and relative emphasis across the text — and cites the lines behind each call. No human grader is needed in either mode.

04 Employer Profile normalized weights

The final weights (a combination of matched level and match strength) form the "Employer Profile" — a numeric signature of what the role requires across all competencies, totaling 100%.

05 Passport Integration & Match student fit score

We intersect the Employer Profile with a Student Passport — the verified competency scores for an individual student. By applying the employer's weights to the student's passport scores, we compute a final Fit Score (0-100), ranking students by how well their specific strengths align with the job's actual demands.

How a single score is computed

The input is compared against all five level descriptions for a criterion. Whichever it most resembles wins — that level's number is the score. It's the same judgment a human grader makes ("this reads like a 4 to me"), done by measuring similarity instead.

The cost story

The expensive component — the LLM — is optional. In the default mode the per-item cost is effectively nothing, because the work is lightweight embedding math. When the LLM is on, we don't even have to run it on every input: we can fire it only on inputs that look ambiguous — a negation cue near a criterion, or two levels nearly tied on similarity — so most items never incur the cost.

Trade-offs we're tracking

Trade-off 1 — the cheap mode can miss negation

With the LLM off, a subtle error — like an input saying a skill is not required — can pass through uncaught, because matching keys off the topic, not the negation. That's an acceptable price when inputs are mostly straightforward or the scores feed a soft ranking someone reviews anyway, and it's exactly the gap the toggle closes when accuracy is worth paying for.

Trade-off 2 — a level alone doesn't rank the metrics; magnitude does

Each competency is assigned the level its job description matches best, but that level alone can't rank competencies against each other. One competency can match the top level (5) on a weak signal while another matches a lower level (4) on a far stronger one — and the second is the one the employer actually values. So the final weight combines both axes: how high the matched level is (the importance band) and how strongly the description engages that competency (the match strength), then normalizes across all five so they sit on one scale before feeding the weighted average.

What's next

The main open question is how to combine the matched level with the match strength into one weight, and how to calibrate similarities across metrics — subtracting each metric's baseline from a set of reference job descriptions — so the cross-metric ranking is trustworthy rather than approximate.
